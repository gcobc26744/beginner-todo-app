# Project Retrospective

## 專案名稱

Beginner Todo App

## 專案定位

這是一個新手前端 side project，目標不是炫技，而是把基本功做完整。專案很小，但包含一個產品會有的基本流程：輸入、驗證、資料更新、畫面更新、狀態切換、資料保存。

## 為什麼值得做

待辦清單看起來很常見，但它很適合練習「前端資料流」。使用者做一個操作，資料會改變，畫面也要跟著改變。如果這件事理解清楚，之後學 React、Vue 或其他框架時，會更容易理解 state、props、component render 的概念。

## 這次用了哪些技術

- 原生 HTML：建立表單、按鈕、清單與語意化結構。
- 原生 CSS：完成卡片式版面、手機版排版、按鈕狀態與完成樣式。
- 原生 JavaScript：使用 DOM API、事件監聽、陣列方法與函式拆分。
- localStorage：讓資料保存在瀏覽器，重新整理後不會消失。
- GitHub：把專案保存到遠端，建立可展示的作品紀錄。

## 開發過程中的取捨

### 沒有使用框架

這次刻意不用 React 或 Vue。原因是這是一個新手基本功專案，先理解原生 DOM 和事件處理，之後再學框架會更有感。

### 沒有使用後端

資料只存在瀏覽器的 `localStorage`，因為這個階段的重點是前端互動。等之後要練全端時，可以再加上 API 和資料庫。

### 沒有做複雜設計

畫面設計保持簡單清楚，重點放在可讀性、可操作性和手機版可用。

## 遇到的困難

### 1. 狀態同步

如果勾選完成只改 checkbox，但沒有更新 `todos` 陣列，下一次重新 render 就會回到舊狀態。解法是讓所有操作都先更新資料，再呼叫 `render()`。

### 2. 空狀態

一開始如果沒有任務，使用者可能不知道畫面是不是壞掉。所以加上 empty state，讓空清單也有清楚回饋。

### 3. 瀏覽器相容性

`crypto.randomUUID()` 很方便，但不是所有情境都穩定，所以補了 fallback id 產生方式。

## 學到的事

- 小專案也需要清楚說明「為什麼做」。
- README 不是裝飾品，它是讓別人理解作品的入口。
- 面試時不能只說「我做了一個 todo list」，要說清楚資料流、技術選擇、困難與解法。
- 把心得寫下來，會讓專案更像作品集，而不是練習檔案。

## 面試時可以講的版本

我做了一個原生 JavaScript 的 Todo App，目標是練習前端基本資料流。使用者可以新增、刪除、完成和篩選任務，資料會存在 localStorage。開發時我把 todos 陣列當成 single source of truth，每次操作都先改資料，再重新 render 畫面，避免 UI 和資料不同步。這個專案讓我理解了 DOM 操作、事件監聽、狀態更新和資料保存的基本流程。
